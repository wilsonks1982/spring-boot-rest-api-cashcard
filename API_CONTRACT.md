# CashCard API Contract

## Overview
The CashCard resource represents a prepaid card account with balance management capabilities. Each card is uniquely identified and automatically associated with the authenticated user who created it.

## CashCard Resource Schema

### Field Definitions

| Field | Type | Required | Constraints | Source | Description |
|-------|------|----------|-------------|--------|-------------|
| `id` | Long | Yes | Auto-generated, Positive | Server | Unique identifier for the cash card (read-only) |
| `cardNumber` | String | Yes | 10 digits only, Pattern: `\d{10}` | Client | Card number must be exactly 10 numeric digits |
| `balance` | BigDecimal | Yes | Min: 0.00, Max: 1,000,000.00, 2 decimals | Client | Current account balance in currency units |
| `owner` | String | Yes | Auto-populated from username | Server | Name of the authenticated user (read-only, set automatically) |

---

## Validation Rules

### cardNumber
- **Constraint**: `@NotBlank` + `@Pattern(regexp = "\\d{10}")`
- **Valid Example**: `"1234567890"`
- **Invalid Examples**:
  - `""` (blank)
  - `null` (null)
  - `"123456789"` (9 digits)
  - `"12345678901"` (11 digits)
  - `"123456789a"` (contains non-digit)
- **Error Message**:
  - Blank: `"Card number must not be blank"`
  - Format: `"Card number must be exactly 10 digits"`

### balance
- **Constraint**: `@NotNull` + `@DecimalMin("0.00")` + `@DecimalMax("1000000.00")`
- **Precision**: 2 decimal places (HALF_UP rounding)
- **Valid Range**: 0.00 to 1,000,000.00
- **Valid Examples**: `0.00`, `100.50`, `999999.99`, `1000000.00`
- **Invalid Examples**:
  - `null` (null value)
  - `-0.01` (negative)
  - `1000000.01` (exceeds maximum)
- **Error Messages**:
  - Null: `"Balance must not be null"`
  - Negative: `"Balance must be non-negative"`
  - Exceeds max: `"Balance must be less than or equal to 1,000,000.00"`

### owner
- **Constraint**: Auto-populated from authenticated username
- **Source**: HTTP Basic Authentication credentials
- **Client Action**: DO NOT provide this field in requests
- **Server Action**: Automatically set to the authenticated user's username
- **Notes**:
  - Always set by the server based on authentication
  - Clients cannot override or modify this field
  - Used for authorization and multi-tenancy (each user only sees their own cards)

## Authentication

- **Method**: HTTP Basic Authentication
- **Credentials**: Username and password required for all endpoints
- **Owner Assignment**: The username from Basic Auth credentials is automatically assigned as the card owner
- **Authorization**: Users can only access/modify their own cards (cards where owner equals their username)
- **Example Header**: `Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ=`

### Example Authentication Flow
```
Client sends request with Basic Auth header:
  Authorization: Basic dGVzdDphYmMxMjM=
  (username: test, password: abc123)
         ↓
Server extracts username: "test"
         ↓
Server automatically sets owner: "test"
         ↓
Card is saved with owner = "test"
```

## API Error Response Format

All error responses follow this standardized format:

```java
record ApiError(
    LocalDateTime timestamp,    // When the error occurred (ISO-8601 format)
    int status,                 // HTTP status code (400, 401, 403, 404, 500, etc.)
    String StatusText,          // HTTP status text (e.g., "Bad Request", "Unauthorized")
    String message,             // Detailed error message
    String path                 // The request path that caused the error
)
```

### ApiError Field Descriptions

| Field | Type | Description | Example |
|-------|------|-------------|---------|
| `timestamp` | LocalDateTime | ISO-8601 formatted timestamp of when the error occurred | `"2026-04-17T11:02:17.091383"` |
| `status` | int | HTTP status code | `400`, `401`, `403`, `404`, `500` |
| `StatusText` | String | HTTP status reason phrase | `"Bad Request"`, `"Unauthorized"`, `"Forbidden"` |
| `message` | String | Detailed, human-readable error message | `"cardNumber: Card number must be exactly 10 digits"` |
| `path` | String | The endpoint path that was requested | `"/cashcards"`, `"/cashcards/1"` |

### ApiError JSON Example
```json
{
  "timestamp": "2026-04-17T11:02:17.091383",
```

## Error Handling Summary

### Common HTTP Status Codes

| Status Code | StatusText | Scenario | Typical Cause |
|------------|------------|----------|---------------|
| 200 | OK | Successful GET/PUT operation | Valid request, resource found and accessible |
| 201 | Created | Successful POST operation | Card created successfully |
| 400 | Bad Request | Validation failure or invalid input | Invalid cardNumber, negative balance, type mismatch |
| 401 | Unauthorized | Missing or invalid authentication | Missing BasicAuth header, invalid credentials |
| 403 | Forbidden | Authenticated but unauthorized to access resource | Accessing another user's card, mismatched owner |
| 404 | Not Found | Resource doesn't exist | Card ID doesn't exist in database |
| 500 | Internal Server Error | Server-side error | Unexpected exception or server failure |

### Error Categories

#### Validation Errors (400 Bad Request)
- Invalid cardNumber format (not 10 digits)
- Blank or null cardNumber
- Null balance
- Negative balance
- Balance exceeds maximum
- Invalid path parameter type (e.g., "abc" instead of number)
- Negative or zero path parameter ID

Example:
```json
{
  "timestamp": "2026-04-17T11:02:17.091383",
  "status": 400,
  "StatusText": "Bad Request",
  "message": "cardNumber: Card number must be exactly 10 digits",
  "path": "/cashcards"
}
```

#### Authentication Errors (401 Unauthorized)
- Missing Authorization header
- Invalid username or password
- Malformed BasicAuth credentials

Example:
```json
{
  "timestamp": "2026-04-17T11:02:17.091383",
  "status": 401,
  "StatusText": "Unauthorized",
  "message": "Invalid authentication credentials",
  "path": "/cashcards"
}
```

#### Authorization Errors (403 Forbidden)
- Accessing another user's card
- Attempting to change card owner
- Request body owner doesn't match authenticated user

Example:
```json
{
  "timestamp": "2026-04-17T11:02:17.091383",
  "status": 403,
```

#### Not Found Errors (404 Not Found)
- Requested card ID doesn't exist
- Card exists but belongs to a different user

Example:

## Data Types & Formats

### Long (id)
- **Format**: Integer
- **Range**: 1 to 9,223,372,036,854,775,807
- **Example**: `123`, `1000`
- **Source**: Auto-generated by server

### String (cardNumber)
- **Format**: UTF-8 encoded text
- **Pattern**: Exactly 10 numeric digits `\d{10}`
- **Encoding**: Should be URL-encoded in requests
- **Example**: `"1234567890"`, `"9876543210"`

### String (owner)
- **Format**: UTF-8 encoded username
- **Source**: Extracted from HTTP Basic Authentication credentials
- **Read-only**: Cannot be modified by clients
- **Example**: `"test"`, `"john_doe"`, `"jane_smith"`

### BigDecimal (balance)
- **Format**: Numeric with up to 2 decimal places
- **Precision**: 2 digits after decimal point
- **Rounding**: HALF_UP (standard banking rounding)
- **Example**: `100.50`, `0.00`, `999999.99`

### LocalDateTime (timestamp in ApiError)
- **Format**: ISO-8601 format with nanosecond precision
- **Timezone**: Server timezone
- **Example**: `"2026-04-17T11:02:17.091383"`

## Multi-Tenancy & Authorization

### Data Isolation
- Each user can only view and modify their own CashCards
- The system enforces this using the `owner` field
- CashCards created by user "test" cannot be accessed by user "admin"

### Access Control Flow
```
Request to GET /cashcards/1 with auth "test:abc123"
         ↓
Extract username: "test"
         ↓
Retrieve card with id=1, owner="test"
         ↓
Check: authenticated username == card owner?
         ├─ YES → Return 200 OK with card data
         └─ NO  → Return 403 Forbidden with ApiError
```

## Implementation Notes

### Balance Rounding
The balance is automatically rounded to 2 decimal places using HALF_UP rounding:
- Input: `100.556` → Output: `100.56`
- Input: `100.551` → Output: `100.56`
- Input: `100.545` → Output: `100.55` (HALF_UP rounds .5 up)

### ID Generation
IDs are auto-generated by the database. Clients should not provide IDs on CREATE operations. IDs are immutable once assigned.

### Owner Field Assignment
On POST (create):
- Client must NOT provide the `owner` field
- Server automatically extracts username from Basic Auth header
- Owner is set to the authenticated user's username

On PUT (update):
- Client should include the `owner` field to match authenticated user
- Server validates that `owner` equals the authenticated username
- Attempts to change owner to a different user are rejected with 403 Forbidden

### No Owner Override
- Clients cannot create cards for other users
- Clients cannot transfer cards to other users
- Attempting to do so results in 403 Forbidden

### Concurrency
No explicit versioning or optimistic locking is implemented. Last-write-wins for concurrent updates.

### Error Response Timestamps
- All error responses include an accurate server timestamp
- Timestamps are in ISO-8601 format with nanosecond precision
- Timestamps help with debugging and auditing requests
